## 核心论断

> **channels 比 jiak 更重要。频道是根，jiak 是叶。频道是不可再生的原始证据。**

一年后回头看，这句话站在哪一边？让我们逐层拆解。

## 一、channels 工程的整体稳固性评级

### 综合评分：B+（足够成为长期基石，但有明确加固项）

```
channels 存储引擎:     A-  ✅ JSONL纯文本是时间的朋友
分子目录隔离:          B+  ✅ 扎实设计，细节欠缺校验
flock并发保护:         B   ⚠️ 本地可靠，NFS/WSL边缘存疑
merge时间线:           B-  ⚠️ 缺少次要排序键+HLC
FTS5索引:              A   ✅ 可丢弃，随时从JSONL重建
波扩散引擎:            B   ⚠️ 功能完整但一年后可能被替代
branch/fork:           B   ✅ 和频道一样是根级能力
jiak集成接口:          C+  ❌ 未实现真正的自动耦合
```

## 二、存储格式生命周期深度分析

### 2.1 JSONL 格式的生存能力：★★★★★（5/5）

ISA v0.6.0 最正确的决策是：

**从 SQLite 迁移到 JSONL。**

原因不是 SQLite 不好，而是：

1. **JSONL 是行格式** — 每行独立、自描述、可互操作。`cat`, `grep`, `head`, `tail`, `wc -l`, `sort`, `jq`, `rg` 都能直接处理。不需要专用驱动，不需要 API 版本匹配，不需要 schema 迁移脚本。

2. **JSONL 没有 schema** — 新字段可以随时添加。旧 reader 看到不认识的字段直接忽略。一年后 v0.7.0 加了 `thread_id` 字段，v0.6.0 的 reader 依然能正常读取旧行。向后兼容性天然满分。

3. **JSONL 没有数据库文件损坏风险** — SQLite 的单文件在 fsync 失败、掉电、多进程并发写时的损坏模式是"整个数据库不可用"。JSONL 的损坏模式是"最后一行可能不完整"。两者都是概率事件，但 JSONL 的退化路径优雅得多——丢失最后一条 vs 丢失整个数据库。

4. **JSONL 是纯文本** — 十年后，就算 Python、SQLite、FTS5 全都过时了，`cat events.jsonl` 仍然展示可读数据。`jq` 能解析。甚至 Notepad 都能打开。**纯文本是唯一真正跨世纪的数据格式。**

### 2.2 JSONL 的退化路径（真实风险）

一年后最大可能的退化场景：

| 场景 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 单文件超过 10GB | 高（如果 channels 被高频使用） | grep/jq 变慢，但仍是流式读取 | 按日期分片已内置（`LOG/2026-06-15.md` 思路） |
| 行内 JSON 嵌套过深 | 低（ISA body 通常短文本） | 读入内存消耗大 | 设置单行大小上限 |
| UTF-8 多字节跨行 | 极低（每行独立 encode/decode） | 无影响 | 已通过 binary append 确保 |
| metadata 膨胀 | 中（`meta` 字段无限制增长） | 行变长，检索变慢 | 建议 v0.7 加 meta 大小上限（默认 4KB） |

**结论**: JSONL 格式一年后完全可读、可迁移。它是 ISA 所有组件中生命周期最长的部分。JSONL 活到 2040 年毫无问题。

### 2.3 flock 的生命周期：★★★（3/5）

flock 是正确的选择但有限定条件：

- **本地 ext4/btrfs/XFS**：一年后仍然可靠。POSIX flock 语义稳定，Linux 内核不会 break 它。
- **NFS/CIFS/SMB**：一年后仍然不可靠。flock 在 NFS 上做空操作是 Linux 已知问题，不是 bug，是设计限制。
- **WSL /mnt/**：flock 在 DrvFs/9P 协议上行为未定义。如果用户把 `~/.hermes` 移到 `/mnt/c/`，flock 保护消失。

**技术债标记**: 建议 v0.7 增加文件系统类型检测，在 NFS/mnt 上启动时输出警告。

### 2.4 FTS5 索引的生命周期：★★★★（4/5）

FTS5 作为索引层是可丢弃的加速器。一年后：

- JSONL 在 → 索引可以重建 → FTS5 层**不是单点故障**
- SQLite FTS5 是 SQLite 内置功能，不会轻易废弃
- 但搜索体验依赖于 FTS5 的中文 FTS5 tokenizer——一年后如果 SQLite 版本更新 break 了 tokenizer，搜索退化到 LIKE 查询（当前 CJK 回退逻辑已处理）

**技术债标记**: 当前 `ingest()` 在每次写入后都调用 `self.index.rebuild()`（第 493 行）。一年后如果频道达到 10 万条信号，每次写入重建索引需要数秒。这是明确的性能技术债。

## 三、哪些设计会成为技术债？

### 🔴 P0 技术债（一年后肯定要修）

**1. merge 时间戳排序缺少次要键**

```python
all_events.sort(key=lambda l: Signal.extract_timestamp(l))
```

同一微秒的两个事件排序不确定。如果设备 A 的消息和设备 B 的回复发生在同一微秒，merge 后可能因果倒置。**这是当前最危险的技术债。**

赫淮斯托斯已给出修复方案（1 行代码加 `device_id` 次要键），建议 v0.6.1 立即修复。

**2. 无混合逻辑时钟（HLC）**

纯 wall clock 在以下场景失效：
- WSL 宿主机休眠/唤醒后时钟偏差可达数秒
- VM 快照回滚后时间倒退
- 树莓派 RTC 电池耗尽后时间跳到 1970 年

当前设计没有 HLC，跨设备 merge 后的时间线在时钟偏差场景下**不可信任**。这不是一年后的问题——如果今天就用多设备，今天就有问题。

**3. `ingest()` 每次写入后全量重建 FTS5 索引**

```python
def ingest(self, signal: Signal) -> str:
    sid = self.store.append(signal)
    self.index.rebuild()  # ← 全量重建！O(n)
    return sid
```

当前测试数据只有几百条，重建瞬间完成。一年后如果频道积累到百万条信号，每次写入触发全量重建将不可接受。需要改为增量索引或延迟批处理重建。

### 🟠 P1 技术债（一年后可能成为问题）

**4. merged store 可写入**

`merge_devices()` 返回的 `ChannelStore` 没有只读保护。用户误写入 merged 目录的数据会在下次 merge 时丢失。这是**静默数据丢失路径**。

**5. device_id 无合法性校验**

```python
DEFAULT_DEVICE_ID = os.environ.get("ISA_DEVICE_ID", "local")
```

攻击者设置 `ISA_DEVICE_ID=../channels/evil` 可导致路径遍历。当前 iterdir 安全，但不应该依赖防御性编程。

**6. 没有信号去重机制**

UUID4 12 位 hex 的碰撞概率 2^-48，但同一信号通过两个设备写入时会出现重复 ID。ISA 的"不可变"哲学应该包含"每个信号全局唯一"的保证——当前没有。

**7. 波扩散引擎的语义距离计算使用 Jaccard（关键词交集）**

Jaccard 距离在关键词少（<5个唯一词）时信息量很低。一年后如果 Agent 数量增加到几十个，波扩散的"附近Agent"检测可能产生大量假阳性。

### 🟡 P2 技术债（一年后可能值得优化）

**8. 缺少频道的日志轮转/归档策略**

JSONL 持续追加没有上限。一年后积累大量旧数据。ISA 使用了按设备分子目录，但没有按时间分片（jiak 的 `LOG/YYYY-MM-DD.md` 设计更好）。当前所有事件在同一个 `events.jsonl` 中。

**9. `read_all()` 会随着频道增长线性变慢**

所有检索操作最终都依赖 `read_all()`（JSONL 全量扫描）。一年后需要：
- 时间范围索引（`read_range` 已实现但只过滤不加速）
- 按页读取
- 冷热数据分层

**10. 无通道级别的加密**

一年后如果 channels 存储敏感对话，纯文本 JSONL 是安全风险。虽然 MVP 明确排除了加密，但需要设计加密层插入点。

## 四、jiak 卡片 vs channels：一年后谁更重要？

> **克洛诺斯断言：channels 比 jiak 更重要。频道是根，jiak 是叶。频道是不可再生的原始证据。**

### 验证

| 维度 | channels（原始证据） | jiak（提炼知识） |
|------|-------------------|----------------|
| **再生能力** | ❌ 不可再生。丢了就没了 | ✅ 可从 channels 重建 |
| **格式寿命** | JSONL → 纯文本 → 无限 | JSON → 纯文本 → 无限 |
| **信息密度** | 低（含所有细节） | 高（仅保留精华） |
| **下游依赖** | jiak、PLUR、搜索都依赖它 | 只有注入依赖它 |
| **一年后风险** | 数据量膨胀 | 结构化骰子误判 |

**结论：克洛诺斯说得对。** channels 是原始记录，jiak 是提炼产物。如果一年后必须二选一保留，保留 channels JSONL，jiak 可以从 channels 重新生成（如果设计好通道→jiak 的自动提取管道）。

### 但有一个关键缺失

当前 channels → jiak 的自动提炼**没有实现**。README 描述了"jiak引擎自动扫描频道LOG，提取决策/观点/问题 → 写入jiak卡片"的美好愿景，但 ISA v0.6.0 的代码中没有这个自动耦合。jiak 和 channels 目前是独立运行的两个系统。

这是 ISA 生态中最大的鸿沟——"频道是原始材料，jiak是提炼后的知识"的闭环需要手动桥接。

## 五、一年后回头看：哪些会被推翻？

### 会被保留的（长期基石）

1. ✅ **JSONL 存储格式** — 生命周期最长的部分。纯文本不可战胜。
2. ✅ **分子目录隔离（按 device_id）** — 多设备方案的根设计。
3. ✅ **append-only 不可变写** — fsync + 追加 = 可信时间线。
4. ✅ **频道 = 目录 = 时间线** — mkdir 语义就是分身语义。
5. ✅ **FlOck 跨进程保护** — 本地部署场景足够。

### 可能被替代或推翻的（中期风险）

6. ⚠️ **FTS5 索引** — 如果搜索量级上升，可能被 Elasticsearch/Lucene 或向量搜索替代。但 JSONL 在，FTS5 可重建，推翻成本低。
7. ⚠️ **波扩散引擎的 Jaccard 语义距离** — Agent 数量从 2 个增长到 200 个时，关键词交集方法可能不够用。需要向量嵌入或 LLM 判据。但 WaveEngine 是纯功能组件，替换不影响存储层。
8. ⚠️ **`branch()` 的时间戳 fork** — 当前 fork 点基于单条 ISO 时间戳。分支演变后合并冲突解决策略未定义。fork 本身正确，但 merge-back 策略一年后可能需要大脑。

### 会被淘汰的（高风险）

9. ❌ **每个写入都全量重建索引** — 见 P0 #3。这是 v0.6.0 为了 MVP 速度做的妥协，生产环境不能这样。
10. ❌ **结构化骰子评分（_score_signal）** — 骰子评分（长度 > 60 字 = +2、含感叹号 = +3）是启发式的。如果一年后有更精确的评分方法，这个函数会被替换。但它只有 10 行，替换成本接近于零。

## 六、工程建议（以一年后视角回溯）

### 如果今天是 2027-06-16，我会希望 2026 年的团队：

1. **在 v0.6.1 修复三个 P0**：merge 次要排序键、HLC、增量索引。这三项是阻碍 channels 成为"长期基石"的最大障碍。
2. **实现 channels → jiak 自动提炼管道**：这是 ISA 生态的最大价值缺口。没有自动提炼，channels 只是日志存储不是知识引擎。
3. **添加频道的存储上限策略**：明确的轮转/归档/删除策略。当前是无限增长。
4. **定义 merge-back 协议**：branch 之后如何合并回主频道？Git 有 rebase/merge 选择。ISA 需要定义"分支时间线的合并语义"。
5. **继续保持超低依赖**：ISA v0.6.0 零外部依赖（纯标准库）是最大优势。一年后回头看，这个决策拯救了多少兼容性问题。

### 长期生存能力指数

| 组件 | 10年生存率 | 理由 |
|------|-----------|------|
| JSONL 格式 | 99% | 纯文本，零依赖 |
| 按设备分子目录 | 95% | 文件系统目录结构，万年后也认得 |
| append-only 不可变写 | 95% | 和 JSONL 共生 |
| flock 保护 | 70% | 如果系统迁移到分布式存储，flock 失效 |
| FTS5 索引 | 60% | 可能被向量搜索替代 |
| 波扩散引擎 | 40% | 需要更复杂的语义距离方法 |
| 结构化骰子评分 | 30% | 启发式方法可能被 ML 方法替代 |

## 七、最终裁决

```
克洛诺斯综合评价: B+（足够成为长期基石，三个 P0 必须加固）
─────────────────────────────────
存储格式:        A+  近乎完美——JSONL是时间的朋友
分子隔离:         A   结构清晰、故障域隔离
flock保护:        B   本地可靠，边界存疑
merge机制:        B-  需要次要键+HLC，否则因果倒置
波扩散:           B   功能完整，方法论一年后可能过时
jiak集成:         C+  缺口：没有自动提炼
索引策略:         C   全量重建是 MVP 妥协，必须改
```

**一句话**: channels 工程的设计哲学（本地优先、文件系统、JSONL、不可变）是经得起时间考验的。技术债集中在 merge 时间线排序和索引维护上，但这些是增量修复，不是架构重构。一年后 channels 能活得好——前提是 v0.6.1 修复 P0。
